AI CHAT SANDBOX - COMPREHENSIVE IMPROVEMENT PLAN
==================================================

ARCHITECTURE SUMMARY
--------------------
The app has a clean Android architecture: Room entities (Chat, Message) with a
single ChatDao, a ChatRepository wrapping the DAO and ApiClient, Hilt DI via
AppModule, DataStore preferences, and Compose UI with navigation. The API layer
uses Retrofit with OkHttp SSE streaming, piped through Kotlin Flows. The
codebase is small (~27 Kotlin files), well-structured, and ready for expansion.


================================================================================
TIER 1: FOUNDATION & DAILY-DRIVER ESSENTIALS (Weeks 1-4)
================================================================================

These fix correctness issues, prevent data loss, and implement features users
will reach for every single session.


1.1 SAFE DATABASE MIGRATIONS (Critical - Day 1)
------------------------------------------------
Problem: AppModule.kt line 25 uses fallbackToDestructiveMigration(). Every
schema change wipes user data.

Implementation:
- Remove fallbackToDestructiveMigration() from AppModule.kt
- Set exportSchema = true in AppDatabase and configure the room.schemaLocation
  KSP argument in build.gradle.kts
- Add a Migration(1, 2) object for every schema change going forward
- Place migrations in a new file data/local/Migrations.kt and register them
  with .addMigrations(MIGRATION_1_2, ...) on the builder
- Bundle all upcoming schema changes into a single MIGRATION_1_2 using
  ALTER TABLE statements

Files: AppModule.kt, AppDatabase.kt, new Migrations.kt


1.2 MESSAGE EDITING & RESEND
-----------------------------
Problem: Users cannot correct a typo or rephrase a prompt without deleting
and retyping.

Implementation:
- Add an "Edit" option to the DropdownMenu in MessageBubble (user messages only)
- When tapped, populate the ChatInputBar text field with the message content
  and set editingMessageId on ChatUiState
- On send while editing: delete all messages after the edited message
  (inclusive of the old assistant reply), update the edited message content
  via ChatDao.updateMessage(), then fire sendMessage() with truncated history
- Add DAO query: DELETE FROM messages WHERE chatId = :chatId AND
  createdAt >= :timestamp

Files: ChatDao.kt, ChatViewModel.kt, ChatScreen.kt


1.3 RESPONSE REGENERATION
--------------------------
Problem: No way to ask for a different response without manually re-sending.

Implementation:
- Add a "Regenerate" icon button below the last assistant message bubble
  (visible only when not streaming and last message is from assistant)
- In ChatViewModel, add regenerateLastResponse(): delete the last assistant
  message, then call sendMessage() using the remaining message history
- Reuses existing streaming logic and composes with stopGenerating()

Files: ChatViewModel.kt, ChatScreen.kt


1.4 RETRY LOGIC WITH EXPONENTIAL BACKOFF
-----------------------------------------
Problem: A single network hiccup kills the request. The streaming fallback
to non-streaming is the only resilience.

Implementation:
- Create data/remote/RetryPolicy.kt with retryWithBackoff() utility
  (maxAttempts=3, initialDelayMs=1000)
- Wrap createChatCompletion() and createChatCompletionStream() calls in
  ApiClient with this retry policy
- Only retry on 429, 500, 502, 503, 529 status codes
- Parse Retry-After header when present
- Surface retry attempts in UI via retryAttempt field on ChatUiState

Files: New RetryPolicy.kt, ApiClient.kt, ChatViewModel.kt


1.5 CONVERSATION SEARCH
------------------------
Problem: No way to find a past conversation or search within messages.

Implementation:
- Add FTS4 support to Room via @Fts4(contentEntity = Message) entity
  called MessageFts
- Add DAO queries joining messages with MessageFts for full-text search
- Add a SearchBar (Material3) at the top of ChatListScreen
- Create SearchViewModel that queries FTS and returns results grouped by chat
- Results show matching snippet with chat title, tapping navigates to chat
  and scrolls to the message

Files: New MessageFts.kt, ChatDao.kt, AppDatabase.kt, ChatListScreen.kt,
       new SearchViewModel.kt


1.6 AUTO-TITLE GENERATION VIA AI
---------------------------------
Problem: Titles are just the first 40 characters of the first message.

Implementation:
- After first assistant response completes, fire a separate lightweight API
  call (cheapest model) with prompt to generate a 3-6 word title
- Add generateTitle() in ChatRepository using a hardcoded cheap model
- Add autoGenerateTitles preference in PreferencesManager (default true)
- Keep the first-40-chars logic as immediate placeholder until AI title arrives

Files: ChatViewModel.kt, ChatRepository.kt, ApiClient.kt, PreferencesManager.kt

Features that make power users say "I cannot go back."


2.1 TOOL USE / FUNCTION CALLING
--------------------------------
Problem: No support for tool/function calling protocols.

A. Data Model Changes:
- Extend ApiMessage to support OpenAI tools format
- Replace simple content: String with content: Any? (String or List of blocks)
- Add tool_calls and tool_call_id fields
- Create ToolDefinition, ToolCall, FunctionCall data classes
- Extend ChatCompletionRequest with tools and tool_choice fields

B. Tool Execution Engine:
- Create data/tools/ToolRegistry.kt singleton mapping names to executors
- Define ToolExecutor interface with suspend execute() method
- Built-in tools: WebSearchTool, CalculatorTool, CurrentDateTimeTool,
  ReadClipboardTool

C. Conversation Loop:
- After receiving assistant response with tool_calls, enter execution loop:
  1. Save assistant message with tool_calls, display with "Tool Call" chip
  2. Execute each tool call via ToolRegistry
  3. Insert tool result messages (role=tool) into conversation
  4. Send updated conversation back to API
  5. Repeat until assistant responds without tool_calls
- Safety limit: maxToolRounds = 10

D. Message Entity Extension:
- Add metadata column (JSON blob) to Message entity for tool call data

E. UI:
- Render tool calls as collapsible cards with AnimatedVisibility
- Add "Tools" toggle in chat settings panel
- Add tool configuration screen in Settings

Files: New ToolModels.kt, ToolRegistry.kt, ToolExecutor.kt, tool
       implementations, ApiModels.kt, ApiClient.kt, ChatViewModel.kt,
       ChatScreen.kt, Message.kt

2.4 CONVERSATION BRANCHING / FORKING
-------------------------------------
Problem: Linear conversations force users to choose a single path.

Implementation:

- ChatDao needs query to find sibling messages by parentId
- ChatViewModel maintains activeBranch map tracking selected branch at each fork
- Displayed messages computed by walking tree following active branches
- This is the single most differentiating power-user feature

Files: Message.kt, ChatDao.kt, ChatViewModel.kt, ChatScreen.kt
